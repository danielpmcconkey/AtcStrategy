### 2. Step 8 — Definition of Success
Wrote `Governance/definition-of-success.md`. Key design decisions:
- **Scope is the job list (Step 10). Success is output equivalence + process quality.**
  The scope manifest seals what jobs are in scope. The definition of success defines what
  "done" means for each job — it's a formula, not a checklist, because outputs are
  discovered at runtime.
- **External modules as code quality proxy.** Rather than trying to assert "is the code good"
  (impractical for 105 jobs), the bar is: external modules are last resort, must evidence
  that standard modules can't do it, scoped to specific use case only.
- **Doc chain consistency requirement (§2.8).** Any code or proofmark config change traces
  backward through BRD → FSD → tests → code. No broken links.
- **Runtime performance explicitly excluded.** If anti-patterns are eliminated, the resulting
  runtime is accepted.
